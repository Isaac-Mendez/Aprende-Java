## Anatomía de un método (Parte 2)

Después de los modificadores de acceso, lo siguiente que podemos encontrar en un método es la palabra reservada `static`.

---

## ¿Qué significa `static`?

Cuando un método o una variable se marca como `static`, quiere decir que **pertenece directamente a la clase**, y no a los objetos creados a partir de ella.

Esto implica:

- No necesitas crear un objeto para llamar al método. (lo veremos mas adelante)
    
- Todas las instancias de la clase comparten ese mismo método o ese mismo valor.  (lo veremos mas adelante)
    
- El método existe **una sola vez en memoria**, asociado a la clase.
    

---

## Ejemplo sencillo

Imagina que tienes una clase con un método que muestra un mensaje general para todo el juego:

`public static void mostrarMensaje() {     System.out.println("Bienvenido al juego"); }`

Como es estático, se puede llamar así:

`Juego.mostrarMensaje();`

No necesitas crear un objeto de la clase `Juego`.

---

# ❗ Entonces… ¿cuándo NO debes usar static?

Cuando el método representa una acción **de un objeto**, por ejemplo:

- comer()
    
- dormir()
    
- correr()
    
- atacar()
    
- saludar()
    
- mover()
    

Estas acciones dependen del estado del objeto (la vida del personaje, su energía, etc.).  
Por eso **NO deben ser static**.

---

# ❗ ¿Y cuándo SÍ usar static?

Cuando el método **no depende de ningún objeto**, sino que es algo general:

- operaciones matemáticas (`Math`)
    
- utilidades (`Helper`, `Utils`)
    
- constantes
    
- funciones que no usan atributos individuales
    

Ejemplos válidos:

`public static int sumar(int a, int b) { ... }  public static void mostrarMenu() { ... }`


## Importante

El modificador `static` **no se usa para acciones que dependen de cada objeto**, como `comer()`, correr, hablar, disparar, etc.  
Esas acciones pertenecen a cada instancia y por eso **no deben ser estáticas**.
